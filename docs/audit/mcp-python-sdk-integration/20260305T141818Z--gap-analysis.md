## Current State
- `telecom-mcp` uses a custom JSON-line stdio protocol (`tool` + `args` request shape).
- Tool contracts, mode gating, write allowlist/cooldown/rate-limits, and envelope are implemented.
- Tool logic already cleanly split into `tools/`, `connectors/`, and `normalize/` layers.
- No dependency on official `mcp` Python SDK yet.

## Desired End State
- Official SDK-powered MCP server registration with protocol-native discovery and invocation.
- Existing telecom business logic reused behind SDK adapters.
- Stdio first, Streamable HTTP second.
- Maintain existing telecom safety model (inspect default, write gates, redaction, audit).

## Gaps and Complexity
| Gap | Complexity | Rationale |
|---|---|---|
| Replace custom stdio protocol with SDK server registration | Medium | Existing tool functions are already centralized, so adapter layer is straightforward |
| Add SDK dependency and package entrypoint alignment | Low | `pyproject.toml` currently has no runtime deps |
| Expose resources and prompts in addition to tools | Medium | Needs new modules and stable URI/prompt contracts |
| Preserve existing envelope as tool payload contract | Medium | MCP `CallToolResult` wrapper must carry deterministic envelope unchanged |
| Add transport abstraction for future Streamable HTTP | Medium | Stdio path exists, HTTP needs routing/security config |
| Introduce SDK-based client tests / contract smoke | Medium | Existing tests are custom-server oriented |

## Risks
- Sandbox/runtime mismatch risk: existing automation expects current json-line protocol.
- Authentication drift risk: current `gh` auth invalid in this environment indicates operator credential inconsistency.
- Transport risk: adding HTTP expands attack surface and requires explicit TLS/CORS policy.
- Concurrency risk: connector blocking behavior under higher parallel tool-call load.
- Schema risk: SDK-native structured outputs could diverge from telecom envelope if adapter not explicit.

## Sectioned Analysis (Prompt-required)
### 1) Transport parity (stdio now, http later)
- Current: stdio custom loop in `server.py`.
- Target: SDK stdio first (`mcp.server.stdio`), then Streamable HTTP.
- Gap: adapter and runtime bootstrap rewrite.

### 2) Tool registry + schema
- Current: dict registry + manual arg checks + envelope wrapping.
- Target: SDK tool decorators/registration with typed args while preserving existing validation and error mapping.
- Gap: thin translation layer needed, not full tool rewrite.

### 3) State management
- Current: in-process cooldown/rate-limit stores (`CooldownStore`, `WindowRateLimiter`).
- Target: keep behavior initially; optional externalized state later (Redis) for multi-instance correctness.
- Gap: no distributed state today.

### 4) Observability
- Current: audit logger + metrics recorder already in server context.
- Target: preserve same correlation_id and audit fields in SDK path.
- Gap: ensure middleware hooks capture all MCP entry/exit points.

### 5) Security
- Current: env-based secret resolution, write mode gates, allowlist/cooldowns.
- Target: keep deny-by-default with explicit enabled tools + redacted logs for any added resources/prompts.
- Gap: must enforce same policy regardless of transport.
